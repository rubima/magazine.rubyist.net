## はじめに

2025/1/18 (土) に開催された[東京Ruby会議12](https://regional.rubykaigi.org/tokyo12/)の運営記録です。東京Ruby会議12は、全部で300人ぐらいが参加し、総支出額320万円ぐらいのカンファレンスでした。

開催にあたって、笹田さんの[東京Ruby会議11の運営記録](https://magazine.rubyist.net/articles/0054/0054-TokyoRubyKaigi11OrganizeReport.html)にはものすごく助けられたので、東京Ruby会議12も次のために記録を残しておこう、という記事です。

東京Ruby会議12 実行委員会のメンバー、それぞれの担当した領域について執筆しています。

## プログラム・企画

### 東京Ruby会議12のコンセプト

書いた人：おしょうゆ

たくさんの思いつきを導入した東京Ruby会議12でしたが、以下の3点はぶれずに設計したつもりです。

- Rubyを「使う」ことに焦点をあてること
- 「会議」であること
- 東京圏のカンファレンスであること

まず最初が、Rubyを「使う」ことに焦点をあてること。これは文字通りで、Rubyの多様な使われかたをたくさん話題にあげる、ということです。Rubyの使い途といえばRails、という状況が長く続いていますが、Rails以外の話題もなるべく取り入れたつもりです。

RubyKaigiはチーフオーガナイザーの松田さんが「[Rubyユーザーズカンファレンスではない](https://speakerdeck.com/a_matsuda/rubykaigi-introduction?slide=25)」と明言していますが、東京Ruby会議12はその真逆をいった、と言えるかもしれません。

2点目が、「会議」であること。トークを聞いてああ面白かった、となるのも良いのですが、やはりその後の議論につながることは強く意識しました。短い一日の中で休憩時間をなるべく長く確保したり、廊下にホワイトボードを大量に設置したのはこれを実現するためでした。ホワイトボードはわりと活用されていたようで何よりです。

そして最後が、東京圏のカンファレンスであること。せっかくの「地域」Ruby会議ですから。東京圏に住むたくさんのRubyistの良い日となることを意識して企画していました。Regional.rb and the Tokyo Metropolisの企画や、一都六県限定のステッカーはこの点から由来したものです。

これらを一語にまとめたのが「Rubyと暮らす」というコンセプトなのでした。いかがでしたか？

振り返ってみると、近年のRubyカンファレンスシーンにないものを埋めようと試みつつも、RubyKaigiやKaigi on Railsに強い影響を受けた会であったように思います。

## スポンサー

### スポンサー設計

書いた人：なっちゃん

カナダからJohn Hawthornさんを基調講演者として招待したい話が出たタイミングで、スポンサー募集することを決めました（8月下旬）。
スポンサープランは以下の3グレードとしました。

- Rubyスポンサー（30万円・3社まで）
- Goldスポンサー（15万円・4社まで）
- Silverスポンサー（5万円・社数制限なし）

実行委員で「どういうプランなら、自分の会社が東京Ruby会議12にスポンサーしやすいか？」という視点から、社数・価格・プラン内容を相談して決めていきました。

- Ruby・Goldスポンサーがすべて集まれば、基調講演の旅費、会場費、録画費用など、カンファレンスに必要な経費をまかなえるようにする
- 会期が1日だけなので、スポンサートーク枠は最大で3本までとする
- ホワイエのスペース的に、ブースは最大7つまでする（Ruby 3、Gold 4になった）
- Ruby・Goldのは、参加者との交流ができるように、ブース確定のプランにする
- スポンサーが集まるか不安だったため、できるだけ早く予算を確定したく、Ruby・Goldは「先着順」で募集する

全体としてバランスの良いスポンサー設計になったと思うのですが、Ruby・Goldスポンサーを先着制にしたのは失敗でした。
スポンサー募集要項を公開した後に、多くの方から「募集要項公開の1週間後に先着順では、会社の稟議が間に合わない」という意見をいただき、抽選制に変更しました。
継続的でないカンファレンスでは、企業が事前に予算を確保していないことが多いため、できるだけ早めにスポンサー募集を開始し、抽選制にするのがおすすめです。

### スポンサー集め

書いた人：なっちゃん

Ruby, Gold のスポンサー候補の企業には、中の知り合いに直接話したり、実行委員がカンファレンスや各Rubyのイベントを巡回して宣伝したり、懇親会などで相談したりといろいろやりました。その結果、Ruby, Goldの応募のほとんどは直接お声がけした企業からでしたが、数社はイベントをきっかけに応募してくれたので、イベントの行脚するのも大事だなと思いました。  

最終的に、Ruby, Goldともに枠数を超える応募がありました。しかし、スポンサーブースを設置できる個数が限られていたため、泣く泣く抽選を行いました。

Silverスポンサーは、最終的に20社集まりました。直接お声がけした企業よりも、イベント経由やSNSきっかけで応募してくれた企業が多く、運営一同とても驚きました。Silverスポンサーの費用は、前夜祭費用、会場装飾、参加者向けのノベルティやおやつなどに活用しました。

### スポンサーとのやりとり

書いた人：なっちゃん

RubyKaigiのオーガナイザーを務めていることもあり、RubyKaigiのスポンサーとのやりとりを参考にしながら進めました。

1社ずつ個別対応するのは負担が大きいと考え、RubyKaigi Teamが作成した [sponsor-app](https://github.com/ruby-no-kai/sponsor-app) を導入しました。全体で27社とのやりとりをすることになったので、導入して正解でした。募集フォームの管理、メールの一斉送信、ロゴの提出・依頼、スポンサーチケットの発行など、スポンサー対応に必要な業務を一元管理できたのがとても良かったです。

sponsor-app は[Fly.io](https://fly.io/)を使ったので、費用は$5/月ぐらいでした。sponsor-appのようにあまりアクセスが多くないサービスをローコストで運用できるメリットは大きかったのですが、5ヶ月で3回ぐらい障害が発生したのがちょっと大変でした。

### アンチボッチランチ

書いた人：いかるが

「前夜祭がやりたいけど人が足りない!!」という理由でオーガナイザーに呼ばれた私ですが、折角なので求められていた仕事以外にも一人で完結する程度の企画を一つ持ち込みたいと考えました。

技術カンファレンス、一人で参加して誰とも喋らずに一人で帰るという体験はしてもらいたくないなという思いはあったのと、ご飯に関してある程度の気持ちを持っていたので、その時手がついてなさそうだったお昼の時間に着目し希望者を対象としたシャッフルランチを企画することにしました。いわゆるアンチボッチランチです。

当初は「お昼に指定の場所にいけばマッチングがされてお昼に行ける」状態を目指すことにしたので以下のことを達成するために動くことにしました。

- 参加者に企画を知ってもらう
- 参加者にお昼どこにいけば良さそうか知ってもらう
- 参加者の組を作ってお見送りする

まず、お昼の選択肢を知ってもらうために[東京Ruby会議12のためのランチ情報、あるいはアンチボッチランチの企画について](https://note.com/tokyorubykaigi12/n/n0d2dcea2c9ba) というnoteの記事を書いてランチ事情とともに企画の事前周知を行いました。また、当日にもお昼の前のコンテンツ紹介のコーナーに時間をいただき、企画を周知しました。午前中のトラックはキーノート一本のみで、ホールには参加者がほぼ全員いたので広く周知ができたかと思います。参加者のシャッフルについては、過去のRubyKaigiで行っていたプランニングポーカーをその場で配ってランチの組を作るというアイディアを大いに参考にさせていただきました。

おかげさまで3,4人のグループを7組作って送り出すことができました。企画当初に7,8組（30人程度）の参加者を見込んでいたのですが、ちょうどよく収まった形となります。

ランチのnote記事についても名札の裏にQRコードでリンクを掲載したことも幸いしてか多くの人に周知ができたのではないかと考えています。これはminamiさんの素敵なアイディアでした。

書いた人：ありたそ

- 川崎駅から徒歩数分

予約した当初は「ストレッチ目標として200名の参加者を気合いで集めるぞ！」という温度感だったため、120名が入れる懇親会会場であれば十分であろうという判断を行ったのでした。

今回の会場は横浜市の[鶴見区民文化センター サルビアホール](https://salvia.hall-info.jp/)でしたが、最初は東京23区のどこかでやるつもりだったんですよ（本当です）。300〜400人の規模のカンファレンスをめざすことにして、ゆとりをもって座れるように500〜800人を収容できるホールを前提に探し始めました。

探し始めてから気がついたんですが、公営のホールといわゆるコンベンションセンターって価格が全然違うんですね……。コンベンションセンターはきれいで会議向けの設備も充実していてすばらしいんですが、100万円前後はかかってしまいます。これはキツい。一方、公営ホールならば15万円前後になるということで、現実的なラインとして公営ホールを中心に探し始めました。

探し始めましたというか、23区の公営ホールは全部見たと思います。時期としては2024〜2025年の冬で考えていたのですが、このときすでに2024年7月末。多くのホールは6〜8ヶ月先の予定を抽選で決めているらしく、一歩間に合っていなかったのでした。抽選を経てなお空いている場合は予約できるのですが、都内の土日は抽選後も空いているということはありませんでした。

と、いうことで、スコープを23区の近郊まで広げたのでした。さいたま市・川崎市・横浜市…… と見ていったところ、今回のホールにたどりつきました（神奈川でも6ヶ月先が空いているホールは希少でした）。横浜市といっても西端、つまりわりと東京側に位置していたのと、[東京Ruby会議10](https://tokyo10.rubykaigi.info/)が千葉市開催だったことにも背中を押され、予約しました。

ところで、会場のキャパシティについて一番気をつかわねばならないのはホールではなくホワイエです。セッション外のいわゆる「廊下」の時間帯は参加者の大半がホワイエに滞在するのですから当然ではあるのですが、このことには随分あとになってから気がつき、少々慌てました（どうにかなってよかったです）。

### 物品手配

書いた人：ありたそ

大きなものから小さなものまで、各種手配を行いました。全体の流れとしては以下です。

チーム内で「XXがやりたい」とミーティングやGitHubのIssueでアイデアが出る
アイデアを実現できる業者を見つけ、見積もりを出し、相場感を把握する
見積もり結果をチームにフィードバックし、予算や工数の観点からやる・やらないを決める

ノベルティのデザイン発注を除くと業者とのやり取りの多くは基本的に自分に集約して行われました。実行委員長や副委員長には東京Ruby会議12の根幹にまつわる部分を担当してもらいたく、事務的な部分は極力巻き取りたいという思いから自然にこのような分担に落ち着きました。

実際に手配したものは、プリクラ/バックボード/ホワイトボード/プロジェクター/事業系ゴミ処理業者/ヤマトの集荷/トランシーバー/イベント保険/キーノートスピーカーのホテル予約/その他当日必要な備品、この辺りでしょうか。

他にも手配できるよう準備はしたけどチームのキャパシティや予算的に厳しく最終的にやらなかった事柄もいくつかあります。追加の音響照明オペレーター/発表のリアルタイム配信機材/駅広告/シャトルバス/高級ネックストラップなどです。

個人的に印象に残った点をいくつか挙げます。

- プリクラの設営、説明書には一人で20分程度でやれますとあったが実際には一人でやるのは本当に難しかった。特に屋根の設置がハードで「絶対ハマらないだろこのネジ」って100回くらいは脳内で叫んでいた。
- カンファレンス実施には大量の荷物が必要になるので、集荷/配送は事前に最寄りの営業所へ相談しに行くべき。設営日に大量の荷物が届くことを伝えていなかったため配送が遅延してしまった......また何十部もの着払い伝票が欲しくなった時、その辺の営業所にフラッと行ってもらおうとしてはいけない。ちゃんと会場付きの担当か、近くの営業所に相談しに行く。
- 今回の東京Ruby会議12の規模では混合廃棄物0.7㎥、ダンボール0.3㎥の廃棄物がでた。また地域Ruby会議で出るようなゴミは事業系廃棄物に分類され、家庭ゴミのように捨てることができない。今回は会場が横浜市だったため横浜市事業系廃棄物対策課に問い合わせを行なった。市が公開している廃棄物処理業者の名簿を共有してくれて、そこから会場から近そうなところに問い合わせを行なった。まずは会場の市区町村役所に相談すると良さそう。
- 前夜祭でクラフトビールを発注したFAM333さんは本当に最高。提供するビール選定の相談に丁寧に乗ってくださるところから始まり、当日はビールサーバーの設営・配布・撤収とすべてをやってくれる。

### 機材レンタル

今回機材のレンタルは[パンダスタジオ](https://rental.pandastudio.tv/)で手配した。
他にもレンタルできる業者はあると思うのでその比較に使えるかもしれない「よかったところ」と「微妙だったところ」を覚えている範囲で列挙しておく。

- よかった:
  - 梱包、発送が丁寧
  - レンタルできる機材が幅広くて便利
- 微妙だった:
  - 回収は「午前中」という指定だったのだけど、これが指す時間が 09:00~14:00 で辛かった。
    - (12:00 までに来ると思ってたけど来なくて不安になって問い合わせをした。ちなみにどこにも 14:00 までを午前中とするという記述は無くて難しい。)
  - 検品がめちゃくちゃ丁寧にされているわけではなさそう。
    - たとえば ZOOM F6 やレンタルした MacBook には前のレンタル者が残したであろうコンフィグが残っている。
    - とくに MacBook はファクトリーリセットすると全損扱いになりうる利用規約になっていて迂闊に消せない。
      - こういうイベントごとで余分に Mac を準備したいとき、良識的な方法は何になるんだろうな〜難しいと思った。

録画録音ではないけど、会場ではスタッフと本屋さん用に wi-fi が準備されていて、これもレンタルしたものだった。
いわゆるポケット WiFi 的なもので、[WiFiレンタル屋さん](https://www.rental-store.jp/)でレンタルした。最速だと秋葉原の店舗で当日受け取れて便利だった。

### 使ってよかった機材

ATEM Mini Extreme ISO、便利でした。これは自明。
録画しないケースでもこれを借りてスイッチングするとよさそう。難しくない。

録画機能がかなり最高で、8入力をバラバラに録画できるだけでなく、それぞれのスイッチングしたタイミングを保持した Davinci Resolve の編集データも生成できる。

もちろん足りない部分もある。1つは画質である。
iPhone の動画ですら時代は 4K だというのに入力と録画の上限サイズは 1920*1080 でやや物足りない。
ライブスイッチャーの世界でこれ以上のサイズを目指すと SDI 入出力の機材になり、SDI-HDMI 変換を大量に準備したりセットアップすることになってお金も熱も電力も大変なことになってしまう。
僕の知識が足りなくて今回は 1920*1080 サイズでの録画録音編集になった。コスパよく 4K 程度のサイズでライブスイッチャーの機能を持ち、録画と録音ができる機材は無いのだろうか…

ZOOM F6 もかなり最高。便利なオーディオインターフェース兼レコーダーとして活躍した。
ただし録音を Atem Mini Extreme ISO と別にしたのが良かったかどうかはわからない。ATEM Atem Mini Extreme ISO に取り込んでおけばそこで生成される Davinci のプロジェクトファイル上でも同期されて便利だったかも…？とも思う。

## 舞台設計と設営

書いた人：てるふの

予約したホールの舞台設備を担当する方とミーティングして、当日舞台上で起きることを整理する会こと舞台打ち合わせが存在する。
舞台打ち合わせでは、↓のことを決める。

- 当日レンタルする備品のリスト
- 当日設置する備品の大まかな位置
- 当日のざっくりとした照明パターン

### 当日レンタルする備品を決める

事前に用意しておいた機材のリストとある程度物理配置と対応がとれた図を用意しておく。ref: 録画録音の設計

この図のうち、どこからどこまでが自前で準備するもので、どこからどこまでが会場の備品で賄って欲しいのかをまずこちらから提示して、その上で実現可能な境目を探して決める。

入力と出力について分けて考えると複雑さが減ってお互い考えやすい。今回↓のような要望を出し、ほぼそのままの境界で機材を準備した。

- 音声録音部分は会場のマイク音声の rec out を用意して XLR ケーブルでオペレーションをしているテーブルまで持ってきて欲しいことを伝えた
- こちらから映像と音声の出力を渡す部分は HDMI ケーブルと RCA のポートまで用意するので、HDMI を指すポートと RCA ケーブルを準備してほしいことを伝えた

今回だと、ギャラリー(3F)については演台とサブスクリーン以外は全部自前だったのでギャラリー側の舞台打ち合わせに相当する打ち合わせはなかった。
ただし、事前に演台をホールから移動することの許可をおらい、サブスクリーン1枚をギャラリーに持ってきてもらうことを会場の担当者と決めておいた。

この打ち合わせで事前に把握してなかった会場備品を知り、いくつか追加で備品をレンタルした。
たとえばギャラリーで使ったサブスクリーン用のプロジェクタは会場の備品をレンタルしたものだった。

### 大まかな位置の決定

自前で準備するもの、借りるもの含めて舞台上にどのように配置するのかある程度決めておく。
当日舞台上で調整はできるので、厳密でなくてもよい。スペース的に無理なく実際に設置できるのかを検討するためのもの。
今回↓のような配置を事前に決めて、当日を迎えた。
当日はサブスクリーンの配置と演者カメラの配置を微調整した。

### ざっくりとした照明パターン

当日ありえる照明パターンをある程度考えておく。
あまりにもパターンが多い場合は準備するものが変わってくるため。
使ったパラメータは、演者スポットライト/ステージ全体/客席の3つで、これを決めておいて当日までに照明の角度や位置をセットアップしてもらう。
明るさは当日容易に変更できるが、角度や配置は変更するのが大変なものもあるので特殊な演出を検討している場合ここで伝える必要がある。

今回は↓の3パターンを設計して舞台袖で操作した。

- 客席25% 演者スポットライト90% ステージ0%
- 客席35% 演者スポットライト0% ステージ90%
- 客席100% 演者スポットライト0% ステージ0%

Regional.rb and the Tokyo Metropolis のときには必要に応じて客電(客席照明)を明るくしたりする対応をした。
今回の会場では↓のようなデバイスが舞台袖にあり、スライダーでシュッと調整ができる。

## 字幕

書いた人：てるふの

ホールとギャラリーには字幕が実装された。
これは Keynote スピーカー含め、海外からの参加者や日本語を母国語としない参加者のための対応だった。
今回の東京Ruby会議12 では登壇者が常に日本語で喋り、それを英語に翻訳したものを文字にしてサブスクリーンに出すものだった。

[https://github.com/tokyorubykaigi12/captioner](https://github.com/tokyorubykaigi12/captioner)

録画録音の項目にある、オーディオインターフェース(ホールは ZOOM F6, ギャラリーは MOTU M2)からの入力を ffmpeg でシュッと Ruby に渡し、AWS SDK を使って AWS Transcribe へ Stream で渡した。
Ref: [https://github.com/tokyorubykaigi12/captioner/blob/main/aws-captioner/transcribe_engine.rb#L31](https://github.com/tokyorubykaigi12/captioner/blob/main/aws-captioner/transcribe_engine.rb#L31)

AWS Transcribe からはちょっとずつ(partial_result)文字起こしされた日本語が返却されるので、これを Claude へ渡して日本語の refine をしてもらった。
Ref: [https://github.com/tokyorubykaigi12/captioner/blob/67931b7a87d5e81643cdd13a037e733e4fc5e65a/aws-captioner/refiner.rb#L22](Ref: [https://github.com/tokyorubykaigi12/captioner/blob/67931b7a87d5e81643cdd13a037e733e4fc5e65a/aws-captioner/refiner.rb#L22)

osyoyu による素晴らしいプロンプトエンジニアリングによって、フィラーはもちろん、ルビーが Ruby になったりして大変良かった。

Refine された日本語を、AWS Translate に逐次投げて英語に翻訳、翻訳結果をいい感じに id を降って AWS AppSync Events に投げるところまでを Ruby で実装し、舞台袖の mac で実行していた。
AWS AppSync Events に投げられた event は、React で実装された Web UI 側で購読し、新しく降ってくる度に id を参照して更新する感じで実装していた。
この Web UI を Chromebook から開いてプロジェクターに渡し、サブスクリーンに投影すれば字幕の完成である。
